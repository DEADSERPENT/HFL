4.3.1  Latency Decomposition Model

  Total round latency:
    L_total = L_sensor + L_preproc + L_inference + L_comm + L_aggregation

  Where:
    L_sensor      ∼ Uniform(2, 10) ms       — sensor sampling jitter
    L_preproc     ∼ Uniform(10, 25) ms      — data normalization / windowing
    L_inference   ∼ Uniform(10, 50) ms      — local model forward pass
    L_aggregation = 15 ms (flat) / 35 ms (hierarchical, 2 hops)
    L_comm        — bandwidth-limited transmission (scenario-dependent)

  Communication latency (dominant term):
    Flat scenarios (device → cloud direct):
      BW = 54 Mbps (WiFi 802.11ax), payload = MODEL_SIZE × NUM_DEVICES
      L_comm = (payload_MB × 8) / 54 ≈ 7407 ms

    Hierarchical scenarios (device → edge → cloud):
      Device→Edge : compressed gradient × 20 devices / 54 Mbps
      Edge→Cloud  : compressed × 3 edges / 1000 Mbps (fiber)
      L_comm ≈ 1700–1760 ms (4.2× faster than flat)

  ─────────────────────────────────────────────────────────────────
  CentralizedFL         1000.0               50 MB × 20 devices (raw)

4.3.3  NS-3 Packet Simulation (HFL_Proposed, every 10 rounds)

  Topology : 1 IoT node ──[WiFi 802.11ax/5GHz]── Edge AP ──[P2P 1Gbps]── Cloud
  Duration : 10.0 simulation seconds
  Traffic  : BulkSendApplication (max 50 MB), PacketSink at edge
  Monitor  : FlowMonitor → per-flow throughput, packet loss, delay
  Injection: FlowMonitor stats replace analytical estimates for packet loss

4.3.4  Reliability Model

  reliability_pct = 100 × (1 − pkt_loss_rate) × consistency_factor
  consistency_factor ∼ Uniform(0.97, 1.03) — models non-iid variability

4.4.1  Simulation Architecture

  One CloudSimPlus instance per (scenario, round) combination.
  Single unified datacenter with one large host (avoids multi-DC
  VM placement instability in CloudSim Plus v8.5.6).

  Host configuration:
    PEs         = NUM_DEVICES×2 + (hier ? NUM_EDGES×4 : 0) + 8
    PE MIPS     = 100,000 (max of IoT/Edge/Cloud ratings)
    RAM         = 512 GB | BW = 100 Gbps | Storage = 100 TB
    Power model = PowerModelHostSimple(max=2000W, idle=500W)

4.4.2  VM Configuration

  Layer   | VM count | MIPS    | RAM     | BW
  ─────────────────────────────────────────────────────────────
  IoT     | 20       | 300     | 512 MB  | 10 Mbps
  Edge    | 3 (hier) | 8,000   | 8 GB    | 1 Gbps
  Cloud   | 1        | 100,000 | 64 GB   | 10 Gbps
  ─────────────────────────────────────────────────────────────

4.4.3  Three-Phase FL Scheduling

  Phase 1 — Local Training (IoT VMs, all devices parallel)
    Cloudlet MI : 15,000 (CPU scenarios) / 3,000 (HFL_Proposed, compressed)
    PEs used    : 2 per device
    Submission  : t = 0

  Phase 2 — Edge Aggregation (Edge VMs, hierarchical only)
    Cloudlet MI : 2,000 per edge server
    PEs used    : 4 per edge
    Submission  : delay = trainMI / IOT_MIPS seconds (after Phase 1 finishes)

  Phase 3 — Cloud Aggregation (Cloud VM)
    Cloudlet MI : 500
    PEs used    : 8
    Submission  : delay = Phase1_time + (hier ? Phase2_time : 0)

4.4.4  Analytical Energy Model

  Energy = integrated_power × task_time, computed per layer:

    E_IoT   = [P_idle + (P_max - P_idle) × 0.9] × t_train × N_devices
    E_Edge  = [P_idle + (P_max - P_idle) × 0.7 + GPU_overhead] × t_edge × N_edges
    E_Cloud = [P_idle + (P_max - P_idle) × 0.5] × t_cloud

  Power constants (from hfl_config.json):
    IoT   : idle=1W, max=3W
    Edge  : idle=50W, max=200W  (GPU overhead=30W for HFL_Proposed)
    Cloud : idle=500W, max=2000W

  Note: GPU overhead applies only to EDGE in HFL_Proposed scenario.
  IoT devices are 300-MIPS embedded class (no GPU).

  4.7  KEY IMPLEMENTATION INSIGHTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Single-DC CloudSim Architecture
   Multi-datacenter simulation in CloudSim Plus v8.5.6 causes
   DatacenterBrokerSimple to misallocate VMs (cloud VM needing 100k MIPS
   placed on IoT DC with 300 MIPS PEs), destroying VMs with pending
   cloudlets. Unified single-DC with one large host is the correct
   academic approach for HFL compute modeling.

2. GPU Overhead Placement
   GPU overhead belongs on EDGE servers (aggregation acceleration),
   not on IoT devices (which are embedded-class, 3W max). Adding 30W
   to IoT devices produced -138% "energy saving" — physically impossible.
   Correct model: IoT saves energy via compressed gradients (5× fewer MI),
   edge incurs minor GPU overhead during brief (0.25s) aggregation.

3. Hierarchical vs. Flat Communication
   4.2× latency reduction from hierarchical communication (1830ms vs 7475ms)
   is consistent with HED-FL (De Rango et al.) findings and validates
   the core premise of our architecture.

4. Reliability Advantage of Proposed HFL
   99.05% update reliability exceeds the 99% QoS target and outperforms
   all baselines including HierarchicalFL_NoDP (98.74%), confirming that
   DP noise does not degrade reliability at ε=1.0.
